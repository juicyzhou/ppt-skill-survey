# 工程化与质量保障

## 1. 推荐的 source-first 工作区

```text
deck-project/
├── brief.md                 # 目标、受众、时长、交付格式
├── content-plan.md          # 故事线与逐页行动标题
├── evidence/                # 数据、论文、引用和来源
├── assets/                  # logo、图片、图标、字体授权说明
├── template/                # 原始 PPTX 模板与使用说明
├── src/                     # JSON/JS/Python/HTML/Markdown 源文件
├── output/                  # PPTX/PDF/HTML
├── renders/                 # 逐页 PNG 与 contact sheet
├── qa/                      # 机器检查与人工评审记录
└── build.md                 # 固定版本、命令、已知限制
```

不要把 `.pptx` 当作唯一源文件。结构化内容、数据、资产来源和构建方式应进入版本控制；生成产物是否提交由团队策略决定。

## 2. 推荐构建流水线

```text
brief → 证据抽取 → 故事线 → 逐页规格 → 生成 → 结构检查
      → 渲染 → 接触表检查 → 单页细查 → 修复 → 复验 → 交付
```

关键门禁：

1. **内容门**：事实、数字、引用和结论经复核。
2. **结构门**：页数、备注、对象、母版、文件完整性正确。
3. **几何门**：无溢出、重叠、越界、极窄文本框和异常换行。
4. **视觉门**：层级、对齐、留白、对比、版式节奏通过人工检查。
5. **兼容门**：目标 PowerPoint 版本、PDF 渲染和至少一个备用查看器通过。
6. **交付门**：源文件、许可证、字体、素材和生成说明齐全。

## 3. 自动检查建议

- 解压 PPTX 并验证 ZIP/OOXML 结构完整。
- 提取全部文本，搜索占位符、乱码、空标题和重复标题。
- 对已知数据做程序化比对，不让 LLM“重写”数值。
- 读取对象坐标，检测越界和明显重叠；对重叠白名单显式记录。
- 渲染每页并生成 contact sheet，先看全局节奏，再放大异常页。
- 对 image-first 产物运行 OCR，核对文字和数据。
- 对 HTML 运行固定 viewport 截图，检查不同浏览器和离线资源。

## 4. 人工评审建议

至少由内容负责人和视觉/交付负责人分别审一次：

- 只读行动标题，是否形成完整论证？
- 每页是否只有一个主要结论？
- 证据是否真正支持标题，而非装饰？
- 10 秒内能否看出页面重点？
- 是否连续多页使用相同卡片布局？
- 是否存在字体过小、图表过密、颜色语义不一致？
- 讲者备注是否帮助演讲而非重复正文？

## 5. 企业落地的 Skill 组合

推荐把职责拆开：

```text
company-brand-skill       品牌、字体、模板、术语、禁用样式
domain-story-skill        学术/咨询/销售/培训的叙事规则
pptx-render-skill         文件生成、模板编辑、图表和备注
deck-qa-skill             结构、视觉、兼容、引用和交付检查
```

这种组合比把所有规则塞进一个超长 `SKILL.md` 更容易维护，也便于替换底层引擎。

## 6. 安全、隐私与许可

- 明确哪些文件会发送到 LLM、搜索、OCR、图像或模板服务。
- 密钥只通过环境/密钥管理器注入，禁止写入 deck 源文件或日志。
- 运行第三方 Skill 时使用最小文件权限和网络权限。
- 对 AGPL、source-available、无许可证仓库和含第三方资产的项目单独审查。
- 记录字体嵌入权、图库许可、AI 生成图片政策和引用来源。
- 对财务、医疗、法律、科研结论等高风险内容保留人工责任人。

## 7. 可重复基准建议

团队应维护至少四个 golden briefs：

1. 中文管理汇报：测试字体、图表、行动标题和高密度页面。
2. 英文产品发布：测试品牌、图片、动效和叙事。
3. 学术报告：测试引用、公式、图表和备注。
4. 模板更新：测试母版、占位符、对象复用和兼容性。

每次升级 Skill、模型或渲染器时重跑，并保存接触表和评分差异。
